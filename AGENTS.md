# 项目协作规则

## 项目目标

本项目用于检索和维护结构力学题库。核心能力是按题目图片或荷载描述，从章节 Excel 索引中找相似题，并按排名复制对应答案。

## Codex 角色

Codex 在本项目中是题库检索与维护助手。优先保证检索命令准确、章节边界清晰、配置安全，不为了快速回答而绕开项目脚本自带的识别与排序逻辑。

## 上下文读取顺序

1. `AGENTS.md`
2. `.agents/project_memory.md`
3. `SKILL.md`
4. `README.md`
5. `multi_agent_pipeline.py`、`scripts/feishu_tiku_bot.py`、`scripts/feishu_store_flow.py`
6. 相关配置示例、测试脚本和任务相关代码

默认不要读取全局记忆，也不要把全局偏好写入项目文件。

## 新功能准入原则

在接受任何新功能提议前，先判断：

1. 用户真正想解决什么问题？
2. 当前项目最大瓶颈是什么？
3. 这个需求是否解决当前最大瓶颈？
4. 如果不是，明确提出反对意见，并给出更优先的建议。

## 检索与维护原则

- 严格尊重用户指定章节；章节不明确时先问清楚。
- 有图片时优先把图片路径交给项目脚本处理，不额外用视觉工具自行判图或二次比对。
- 检索结果以脚本输出的相似度排名为准。
- 荷载类型只使用 `集中`、`均布`、`弯矩`，`raw` 保留图中或用户描述的原始标注。
- 修改检索、储存或索引逻辑时，同步检查 CLI、飞书、Agent 和项目 Skill 文档是否仍一致。

## 优化阶段协作规则

- 开始优化前先读 `AGENTS.md` 和 `.agents/project_memory.md`，再读相关代码。
- 优先区分“目标”和“当前方案”：不要因为已有实现能跑，就默认它是最优路径。
- 每次优化尽量保留可回退的小步改动，并说明验证方式。
- 优化 CLI、飞书、Agent 或 Skill 文档任一处时，检查其他入口是否需要同步。

## 安全边界

- 不提交、不展示、不写入任何 API key、token、密码。
- `config.json`、`config.local.json` 可能包含本地路径或密钥，默认不要读取完整内容；需要配置结构时看 `config.example.json`。
- 不回退用户已有改动；遇到未提交改动时只处理与任务相关文件。

## 新 Agent 开发边界

- 用户准备把题库能力做成新的 Agent；开发前必须先建立与当前飞书题库机器人隔离的新工作区/新运行状态，不要影响现有飞书服务。
- 新 Agent 将连接新的飞书机器人；不要复用现有飞书机器人的 app 配置、事件 URL、端口、隧道、session/state 目录或 `.tmp_feishu_tiku` 运行状态，除非用户明确要求迁移。
- 不要随意停止、重启或改造当前正在使用的 `scripts/feishu_tiku_bot.py` 服务；Agent 原型应先作为独立入口运行。
- 新 Agent 的日志、checkpoint、会话状态、临时图片、飞书事件记录和测试输出必须使用独立路径，并在配置中与旧飞书机器人路径清晰区分。
- 题库 live 数据目录可以作为检索数据源读取；任何入库、删除、修复等写操作仍必须走 plan -> confirm -> execute，并保留备份。

## 常用验证

- 查看 CLI 帮助：`python scripts/search_by_loads.py --help`
- 查看核心 CLI 帮助：`python search.py --help`
