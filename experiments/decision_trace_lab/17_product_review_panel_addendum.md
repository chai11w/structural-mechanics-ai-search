# 旁路评审面板：复核状态与排序补充规格

## 0. 范围与优先级

本文补充 `16_product_trace_ui_addendum.md`，只调整右侧旁路评审 UI，不修改主线 Agent、左侧搜题流程、trace 采集、标签协议或汇总口径。

冲突时，本文关于复核卡片状态、排序和点击反馈的要求优先。

## 1. 要解决的问题

当前只显示“已复核数量”，用户无法快速确认：

- 具体哪一项已经复核；
- 选择的是正确、错误还是不确定；
- 点击是否已经生效；
- 已复核项占据上方空间，导致不断滚动寻找未复核项。

目标是让用户通过按钮选中状态直接判断当前选择，并优先处理未复核项。卡片不额外显示“已复核：xx”等状态文字。

## 2. 每张卡片只用按钮绿色表示当前状态

### 2.1 未复核

未复核卡片显示：

- 三个可点击按钮：`correct（正确）`、`incorrect（错误）`、`uncertain（不确定）`；
- 三个按钮均为普通未选中样式，没有任何绿色按钮；
- 不额外显示 `unlabeled（未复核）` 状态徽标。

### 2.2 已复核

已复核卡片必须同时显示：

- 当前选中的按钮变为绿色；
- 非选中按钮仍可点击，用于修改标签；
- 不额外显示“已复核：correct/incorrect/uncertain”等状态文字或徽标；
- `revision（修订版本）` 可以只在展开详情中显示，不占用卡片主视图；
- 若填写过 expected/reason，提供“查看说明”入口。

三个 verdict 的选中状态统一使用绿色，不根据“正确/错误/不确定”改成三套颜色：

| 当前选择 | 按钮显示 |
|---|---|---|
| correct | `correct（正确）`按钮变绿 |
| incorrect | `incorrect（错误）`按钮变绿 |
| uncertain | `uncertain（不确定）`按钮变绿 |
| 未复核 | 三个按钮都不变绿 |

按钮本身始终保留 `correct（正确）`、`incorrect（错误）`、`uncertain（不确定）`文字；绿色只表示“当前选中”，不表示语义上的好坏。

## 3. 排序规则

评审队列默认分为两段：

```text
待复核
  未复核卡片……

已复核
  已复核卡片……
```

规则：

1. `unlabeled（未复核）` 永远置顶。
2. 已有最新标签的卡片沉到“已复核”区域。
3. 未复核组内部按原始 trace sequence 升序。
4. 已复核组内部也按原始 trace sequence 升序，避免每次刷新乱序。
5. 用户成功提交第一个标签后，该卡片从“待复核”移动到“已复核”区域。
6. 修改已复核卡片的标签时，它继续留在“已复核”区域；新按钮变绿，旧按钮取消绿色，不跳回顶部。
7. 技术详情仍按完整原始 sequence 展示，不受人工队列排序影响。

顶部计数改为：

`待复核 X · 已复核 Y · 共 Z 个关键项`

计数只是摘要，不代替卡片状态。

可增加 `全部 / 待复核 / 已复核` 三个轻量筛选项，但默认使用“全部”，且仍按未复核在前、已复核在后排序。

## 4. 点击反馈

### 4.1 提交过程

用户点击 verdict 后必须立即得到可见反馈：

1. 被点击按钮进入 `正在保存…` 状态或显示 spinner；
2. 当前卡片的三个 verdict 按钮临时禁用，防止重复提交；
3. 不锁定其他卡片，也不影响左侧搜题；
4. 服务器成功后显示短暂的 `已保存` 提示；
5. 新选择的按钮变绿，原来选中的按钮取消绿色，并更新 revision；
6. 首次复核的卡片随后移动到已复核区域。

卡片移动应保持当前页面可理解：可以使用很短的过渡动画，或显示“已移至已复核”提示；不得让卡片无提示突然消失，使用户误以为保存失败。

### 4.2 提交失败

如果标签 API 失败：

- 卡片保持原位置和原 verdict；
- 恢复按钮可点击状态；
- 显示明确错误，如 `保存失败，请重试`；
- 不增加已复核计数；
- 不生成前端假 revision；
- 不影响左侧 Agent 结果或其他卡片。

前端可以做“等待确认”的临时样式，但最终状态必须以后端返回的最新 label 为准，不能仅靠乐观更新永久生效。

## 5. incorrect 的交互

点击 `incorrect（错误）` 时：

1. 展开同卡片内的简短表单；
2. `expected（期望结果）`、`reason（错误原因）` 和 error_category 仍为可选；
3. 用户可以直接保存“错误”，也可以补充说明后保存；
4. 表单提交期间使用上述统一保存反馈；
5. 已复核为 incorrect 的卡片再次展开时，显示当前最新说明。

不得因为 expected/reason 未填而阻止用户记录错误。

## 6. 修改已有标签

标签仍采用 append-only revision，不覆盖历史。

行为要求：

- 用户点击另一个 verdict，即视为修改当前标签；
- 保存成功后，后端追加新 revision；新按钮变绿，旧按钮取消绿色；
- 旧 revision 保留在 labels.jsonl，但 UI 默认只显示最新标签；
- 可在详情中显示“已修改 N 次”，MVP 不要求展示完整历史；
- 用户点击已经选中的 verdict 时，不重复写相同 revision；可保持不变，或仅打开说明编辑区；
- 修改 expected/reason 也必须追加 revision；
- 修改失败时继续显示原最新标签。

NO_MATCH 回合级标签遵循相同规则：当前分类按钮变绿；修改 reasonable/false/uncertain 时，新按钮变绿、旧按钮取消绿色，追加 revision，并保持在已复核区域。

## 7. 刷新与多回合一致性

- 页面刷新后，从后端 latest labels 恢复每张卡片的绿色选中按钮、revision 和排序。
- 不依赖 localStorage 判断是否已复核。
- 切换 turn 时，每个 turn 独立计算未复核/已复核。
- 同一 target+dimension 只用最高 label_revision 作为当前状态。
- 如果标签目标已不存在或事件未加载，不能错误套到另一张卡片。

## 8. 成功标准

以下场景全部通过才可验收：

1. 三个按钮都未变绿表示未复核；唯一绿色按钮表示当前 verdict，不出现额外“已复核：xx”文字。
2. 首次点击 correct 后，按钮立即显示保存中；成功后 correct 变绿、出现短暂已保存提示，卡片沉到已复核区域。
3. 点击 incorrect 后可不填说明直接保存，也可填写 expected/reason。
4. 已标 correct 的卡片改为 uncertain 后，revision 增加，uncertain 变绿、correct 取消绿色，卡片仍在已复核区域。
5. 再次点击当前已选 verdict 不产生无意义的重复 revision。
6. API 失败时卡片不移动、不改计数、不丢失原标签，并可重试。
7. 多张卡片中所有未复核项始终位于已复核项之前；组内顺序与 trace sequence 一致。
8. 刷新页面后绿色选中按钮、revision 和排序保持一致。
9. NO_MATCH 当前分类通过绿色按钮清晰可见且可修改。
10. 标签交互失败或侧栏崩溃不影响左侧主线搜题。

## 9. 明确不做

- 不增加“必须全部复核”的完成门槛。
- 不自动给未复核项打标签。
- 不因卡片沉底而改变原始 trace sequence。
- 不删除旧 revision，不把 append-only 改成覆盖写。
- 不修改主线 Agent、网页左侧、工具、状态或响应。
- 不做批量全选正确、自动判分或强制复核流程。

一句话口径：

> 卡片不增加状态文字：没有绿色按钮就是未复核，唯一绿色按钮就是当前选择；未复核永远优先出现，保存成功后再沉底。
